CHEAPER ALTERNATIVES FIX
========================

DATE: 2026-06-26
STATUS: ✅ COMPLETE

PROBLEM
-------

The "cheaper alternatives" intent was not correctly filtering products.
Products more expensive than the source product were appearing in results.

REQUIRED BEHAVIOR
-----------------

When user asks for:
- "cheaper alternatives"
- "lower-priced alternatives"  
- "budget alternatives"
- "more affordable alternatives"

Then returned products MUST satisfy:
  alternative.price <= source_product.price

This is a HARD CONSTRAINT.

Products more expensive than source product must NOT appear.

EXAMPLE
-------

Source Product:
  Hiking Backpack Arpenaz 500 - ₹1999

Candidates:
  ₹1299 ✅ (cheaper - include)
  ₹1999 ✅ (same price - include)
  ₹2599 ❌ (more expensive - exclude)
  ₹3999 ❌ (more expensive - exclude)
  ₹4999 ❌ (more expensive - exclude)

Only return: ₹1299, ₹1999

CHANGE MADE
-----------

**File**: backend/tools/alternatives_tool.py
**Line**: ~118 (lower_price filter)

**Before**:
```python
if apply_lower_price:
    candidates = [
        c for c in candidates
        if float(c.get("price") or 0.0) < source_price  # STRICT less than
    ]
    logger.info(f"Filter lower_price < {source_price} applied")
```

**After**:
```python
if apply_lower_price:
    # Hard constraint: price <= source_price
    candidates = [
        c for c in candidates
        if float(c.get("price") or 0.0) <= source_price  # Less than or EQUAL
    ]
    logger.info(f"Filter lower_price <= {source_price} applied: {len(candidates)} candidates remaining")
```

KEY CHANGE
----------

Changed from `<` (strictly less than) to `<=` (less than or equal to).

This allows products at the SAME price as the source product.

Rationale:
- Product at same price is not more expensive
- User asking for "cheaper" wants "not more expensive"
- Same price = equally affordable = valid alternative

ISOLATION
---------

The change ONLY affects the cheaper alternatives flow.

The filter is applied AFTER retrieval, BEFORE ranking.

Flow:
1. Retrieve candidates (existing logic - unchanged)
2. Apply constraint filters:
   - If lower_price flag set → filter by price <= source_price ✅ CHANGED
   - Other constraints (same/different brand, rating, etc.) - unchanged
3. Rank remaining candidates (existing logic - unchanged)
4. Return top 10

NO EMPTY RESULTS FALLBACK
--------------------------

If no cheaper alternatives exist:
- Do NOT return more expensive products
- Return empty alternatives list
- Message: "No cheaper alternatives were found in the knowledge base."

This is handled by the existing code - when candidates list becomes empty
after filtering, the response naturally returns an empty products list.
