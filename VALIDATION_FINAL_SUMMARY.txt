================================================================================
VALIDATOR PROMPT (FINAL)
================================================================================

Decision Rules:
RELEVANT: The candidate IS the requested product type.
RELATED: The candidate is an accessory, attachment, storage item, replacement 
         part, or complementary product used WITH the requested product.
NOT_RELEVANT: Different product type.

Example (Requested: golf club):
  Golf club set → RELEVANT
  Golf driver → RELEVANT
  Golf stand bag → RELATED (storage FOR clubs)
  Golf glove → RELATED (worn WHILE using clubs)
  Golf umbrella → RELATED (used ON golf course)
  Tennis racket → NOT_RELEVANT

Example (Requested: backpack):
  Hiking backpack 30L → RELEVANT
  Backpack rain cover → RELATED (protects backpack)
  Water bottle → RELATED (carried IN backpack)
  Tent → NOT_RELEVANT

Output ultra-compact JSON:
{"id1":["DECISION",conf],...}

Format:
{"8828088":["RELEVANT",1.0],"8787845":["RELATED",0.8],"8917578":["NOT_RELEVANT",1.0]}

Return valid JSON only, no formatting.

================================================================================
WHY THESE DECISIONS
================================================================================

1. WHY ULTRA-COMPACT JSON (NOT LINE FORMAT)?
   - Model was fine-tuned on JSON output
   - Custom formats fight against learned distribution
   - JSON is reliable and robust
   - Parser is simple (one-liner core logic)

2. WHY KEEP CONFIDENCE?
   - Used in budget_optimizer.py: validation_score = product.validation_confidence
   - Used in hybrid_search.py: new_score = (hybrid_score * 0.7) + (confidence * 0.3)
   - Essential for scoring and re-ranking

3. WHY KEEP DESCRIPTIONS (TRUNCATED)?
   - Product names like "MH500" or "NH100" are ambiguous
   - Descriptions provide essential context
   - Truncated to 150 chars to balance accuracy and token count

4. WHY EXAMPLES IN PROMPT?
   - LLMs respond extremely well to examples
   - Shows concrete cases of RELEVANT vs RELATED
   - Two examples (golf club + backpack) generalize well
   - Parenthetical explanations make reasoning explicit

5. WHY NOT CHANGE RETRIEVAL?
   - Retrieval is working correctly (broad matching)
   - Validator separates primary (RELEVANT) from accessories (RELATED)
   - Frontend can show them separately or filter as needed
   - Better user experience than filtering at retrieval time
