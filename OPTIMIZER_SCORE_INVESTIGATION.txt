BUDGET OPTIMIZER SCORE INVESTIGATION
=====================================

DATE: 2026-06-26
ISSUE: Optimizer scores are null or identical, causing suboptimal selections

ROOT CAUSE IDENTIFIED
---------------------
The optimizer was maximizing "recommendation_score" which is always null.
recommendation_score is a Phase 2 feature that's not yet implemented.

The optimizer should actually maximize a COMPUTED score based on:
- similarity (hybrid search score)
- validation_confidence  
- profile_score
- budget efficiency

WHAT THE OPTIMIZER ACTUALLY USES
---------------------------------

The optimizer computes its own score for each product using _score_product():

Formula:
  optimizer_score = 0.4 * similarity
                  + 0.2 * validation_confidence
                  + 0.2 * profile_score
                  + 0.2 * budget_efficiency

Component Scores:
1. similarity = Hybrid search score (semantic + keyword matching)
   - Comes from hybrid_search.py
   - Range: 0.0 to 1.0
   - Measures how well product matches user's search query

2. validation_confidence = LLM validation confidence
   - Comes from validation_service.py
   - Range: 0.0 to 1.0
   - Measures confidence that product matches expected type

3. profile_score = User profile match
   - Computed by task_tool.py
   - Range: 0.0 to 1.0
   - Measures match to user profile (kids/beginner/etc)

4. budget_efficiency = Price-based component
   - If budget_sensitive: Lower price = higher score
   - If premium_intent: Higher price = higher score
   - Otherwise: Neutral (0.5) or budget-aware

Storage:
- Scores computed by _score_product()
- Stored in self._product_scores dictionary
- Keyed by product_id
- Retrieved via self._get_score(product)

IMPORTANT: recommendation_score field on Product is NOT USED!
It's a Phase 2 feature and is currently always null.

   - Detects any mismatches

DEBUGGING WORKFLOW
------------------

After restarting api_swagger.py, look for these log sections:

STEP 1: Check Score Computation
================================
Look for: "COMPUTING OPTIMIZER SCORES FOR ALL PRODUCTS"

For each product, verify:
✓ similarity is NOT null (should be 0.0-1.0)
✓ validation_confidence is NOT null (should be 0.0-1.0)
✓ profile_score is NOT null (should be 0.0-1.0)
✓ COMPUTED OPTIMIZER SCORE varies across products

RED FLAGS:
❌ All products have same optimizer score
   → Check if similarity/validation/profile are all the same
   → This would cause tie-breaker to just pick first/cheapest

❌ All scores are exactly 0.5 or 0.6
   → Likely using default values for missing fields
   → Check hybrid_search or validation_service

❌ Similarity is null for all products
   → Hybrid search not computing scores
   → Check hybrid_search.py final_score assignment

DIAGNOSTIC SCENARIOS
--------------------

SCENARIO 1: All products have score 0.6
Cause: similarity is null for all products
Fix: Check hybrid_search.py - ensure final_score is assigned to products

SCENARIO 2: All products have same score
Cause: Component scores (similarity/validation/profile) identical
Fix: Verify ranking is working in hybrid_search and task_tool

SCENARIO 3: Optimizer chooses cheapest despite varying scores
Cause: Tie-breaker not working or scores computed incorrectly
Fix: Verify _find_best_combination logic

SCENARIO 4: item.recommended not matching optimizer's choice
Cause: Assignment logic broken in optimize() method
Fix: Check "BUILDING RESPONSE" section for mismatches
