================================================================================
OPTIMIZATIONS APPLIED
================================================================================

1. ✅ REDUCE CANDIDATE COUNT
   Before: Validates all 25 candidates
   After:  Validates only top 10 candidates
   
   Rationale:
   - Hybrid search is excellent (semantic scores consistently 0.60-0.65+)
   - Top 10 candidates are sufficient for final selection
   - 2.5x reduction in validation work
   
   Code: candidates = req.candidates[:10]

2. ✅ COMPACT JSON OUTPUT FORMAT
   Before (verbose):
   {
     "validated": [
       {
         "id": "8828088",
         "decision": "RELEVANT",
         "confidence": 1.0
       }
     ]
   }
   
   After (compact):
   {
     "validated": {
       "8828088": "RELEVANT",
       "8787845": "RELATED:0.8"
     }
   }
   
   Token Savings:
   - Verbose: ~15 tokens per product
   - Compact: ~8 tokens per product
   - 47% reduction in output tokens
   
   Confidence Handling:
   - If confidence = 1.0: "RELEVANT"
   - If confidence < 1.0: "RELATED:0.8"
   - Parser converts back to array format for backward compatibility

3. ✅ COMPACT PROMPT
   Before: 850+ chars with verbose rules and JSON schema
   After:  ~400 chars with concise instructions
   
   Changes:
   - Removed verbose explanations
   - Simplified decision criteria
   - Showed compact output format by example
   - Removed unnecessary formatting instructions

4. ✅ INCREASED max_tokens (from 512 to 768)
   Rationale:
   - 512 was too aggressive and could truncate output
   - 768 provides comfortable margin
   - Compact format: 10 products × 8 tokens = 80 tokens (well within limit)
   - Not 1024 or 2048 (wasteful)

5. ✅ KEPT CONFIDENCE FIELD
   Important: confidence CANNOT be removed
   
   Used in:
   - backend/tools/budget_optimizer.py (line 524)
     validation_score = product.validation_confidence
     
   - backend/services/hybrid_search.py (line 1015)
     new_final_score = (hybrid_score * 0.7) + (confidence * 0.3)
   
   Confidence is essential for scoring and re-ranking.

6. ✅ KEPT DESCRIPTIONS (truncated to 150 chars)
   Rationale:
   - Product names alone are insufficient for ambiguous products
   - Examples: "MH500", "NH100", "MT100" (need description for context)
   - 150 chars provides sufficient context while reducing token count

7. ✅ DETAILED TIMING INSTRUMENTATION
   Logs 3 phases:
   - Prompt construction (ms)
   - Model generation (seconds)
   - JSON parsing (ms)
   
   Output format:
   ================================================================================
   VALIDATION COMPLETE
     Prompt: 15.2 ms (0.3%)
     Generation: 1.85s (99.2%)
     Parsing: 8.5 ms (0.5%)
     TOTAL: 1.87s for 10 products
     Per-product: 187 ms/product
   ================================================================================

for backward compatibility:

    "8787845": "RELATED:0.8",
    "8917578": "NOT_RELEVANT"
  }
}

Output from endpoint (converted):
{
  "validated": [

    {"id": "8917578", "decision": "NOT_RELEVANT", "confidence": 1.0}
  ]
}

No changes required in calling code (backend/services/validation_service.py).
