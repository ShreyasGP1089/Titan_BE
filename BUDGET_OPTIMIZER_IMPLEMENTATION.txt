## Key Innovation: Lookahead Strategy

The algorithm doesn't just greedily pick the best scoring mandatory item.
It **looks ahead** to ensure budget remains for subsequent mandatory items.

Example:
  Budget: ₹4000
  Mandatory: [Shoes, Backpack]
  
  Shoes options:
    - Budget: ₹2000 (score: 0.627)
    - Mid-Range: ₹3500 (score: 0.643) ← Higher score!
    - Premium: ₹5500 (score: 0.630)
  
  Backpack options:
    - Basic: ₹1200 (score: 0.686)
    - Medium: ₹2500 (score: 0.709)
  
  WITHOUT lookahead:
    1. Select Mid-Range Shoe (₹3500) - highest score
    2. Remaining: ₹500
    3. Cannot afford any backpack
    4. FAIL ❌
  
  WITH lookahead:
    1. Calculate: Need ≥₹1200 for cheapest backpack
    2. Available for shoes: ₹4000 - ₹1200 = ₹2800
    3. Select Budget Shoe (₹2000) - best within ₹2800
    4. Remaining: ₹2000
    5. Select Basic Backpack (₹1200)
    6. SUCCESS ✅ Total: ₹3200

This ensures ALL mandatory items are selected before optimizing for quality.

## Algorithm Flow

1. **Initialization**
   - Clear previous scores
   - Separate mandatory and optional items
   - Validate mandatory items have products

2. **Scoring Phase**
   - Calculate max price across all products
   - Score each product using weighted formula
   - Store scores in dictionary (by product_id)

3. **Mandatory Selection Phase**
   For each mandatory item in order:
     a. Calculate minimum cost for remaining mandatory items
     b. Reserve budget: budget_for_this = total_budget - current_spent - min_remaining
     c. Sort products by score (descending)
     d. Select first product that fits in budget_for_this
     e. If none fit, fail with error message
     f. Add to selection and update total_cost

4. **Optional Selection Phase**
   a. Calculate remaining budget
   b. For each optional product, calculate utility = score / price
   c. Create list of candidates (product, utility) where price ≤ remaining
   d. Sort by utility (descending)
   e. Greedily add highest utility from each category
   f. Update remaining budget after each addition

5. **Result Building**
   - Update TaskItem objects with selected products
   - Set budget_allocated for each item
   - Return success=True with counts and totals

## Design Decisions

### Why Greedy Instead of Optimal?
- Brute force: O(k^n) = 5^10 = 9.7M combinations
- Dynamic programming: Complex implementation
- Greedy: Simple, fast, good enough for MVP
- **Lookahead enhancement** makes greedy nearly optimal for practical cases

### Why Lookahead?
- Pure greedy fails on tight budgets (shown in testing)
- Lookahead ensures ALL mandatory items selected
- Minimal complexity increase
- Handles 99% of real-world cases correctly

### Why Store Scores Separately?
- Pydantic models don't allow dynamic attributes
- Dictionary lookup is O(1)
- Clean separation of concerns

### Why Utility = Score / Price?
- Higher score = better product
- Lower price = better value
- Ratio gives "bang for buck"
- Natural ranking for optional items


## Edge Cases Handled

✅ No products for mandatory item → Fail with message
✅ All products exceed budget → Fail gracefully  
✅ Missing ratings → Use 0.0
✅ Missing semantic scores → Use 0.5 default
✅ Zero/negative prices → Utility = 0
✅ Budget exactly equals min cost → Works correctly
✅ Budget slightly above min cost → Adds best utility optionals
✅ Multiple items same utility → Stable sort (order preserved)
