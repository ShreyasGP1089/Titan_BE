COMPARISON TOOL: INDEPENDENT PRODUCT RESOLUTION FIX
===================================================

DATE: 2026-06-26
ISSUE: Comparison returning unrelated products instead of requested items

PROBLEM STATEMENT
-----------------
Query: "Compare Kiprun KD900X LD+ and Kiprun KD500 2"
Expected: Two Kiprun running shoes
Actual: Unrelated KIPRUN products (T-shirt, Visor, etc.)

ROOT CAUSE
----------
The comparison tool was using a flawed resolution approach:
1. Extracted keywords from product mentions (e.g., ["kiprun", "kd900x", "ld+"])
2. Searched database using those keywords
3. Could match ANY product with those keywords
4. Often matched unrelated products from same brand

The issue: Keyword extraction lost context. "Kiprun" alone could match
any Kiprun product (shirts, visors, accessories), not just the specific
running shoe model requested.

SOLUTION
--------
Complete rewrite to use Product Discovery pipeline for resolution.

NEW FLOW:
User Query: "Compare Kiprun KD900X LD+ and Kiprun KD500 2"
     ↓
Parse comparison request (api_swagger.py)
     ↓
Extract product mentions:
  1. "Kiprun KD900X LD+  "
  2. "Kiprun KD500 2"
     ↓
For EACH mention independently:
  → Call Product Discovery with FULL mention
  → Use hybrid search (semantic + keyword + validation)
  → Select top RELEVANT product
     ↓
Return resolved products for comparison

KEY PRINCIPLE:
Each product mention is a mini-search query.
Use the FULL Product Discovery pipeline, not keyword extraction.

IMPLEMENTATION
--------------

File: backend/tools/compare_tool.py

1. Updated execute() method
   - Enhanced logging to show parsed product mentions
   - Calls new _resolve_product_via_discovery() for each mention
   - Logs resolution results for each product

2. Added _resolve_product_via_discovery() method
   - Takes FULL product mention as input
   - First checks if it's a direct product ID
   - Otherwise, calls Product Discovery (HybridSearchService)
   - Uses full mention as keywords (not extracted subset)
   - Returns top RELEVANT product (not RELATED)
   - Comprehensive logging at each step

3. Kept _resolve_product() method
   - Legacy method for backward compatibility
   - Not used by compare tool anymore
   - May be used by other tools (alternatives_tool)

CODE CHANGES
------------

OLD APPROACH:
```python
def _resolve_product(self, p_ref: str):
    # Extract keywords
    words = p_ref.split()
    
    # Search with keywords
    candidates = search_by_keywords(words)
    
    # Score and pick best
    best = max(candidates, key=score)
    return best
```

Problem: Loses context, matches generic products

NEW APPROACH:
```python
def _resolve_product_via_discovery(self, product_mention: str, mention_index: int):
    # Check if direct product ID
    if is_product_id(product_mention):
        return get_by_id(product_mention)
    
    # Use Product Discovery with FULL mention
    keywords = product_mention.split()  # Keep ALL words
    
    result = search_service.search(
        sport=None,           # Search all sports
        category_level_1=None,  # Search all categories
        keywords=keywords,    # Use FULL mention
        top_k=5,
        return_format='dict'
    )
    
    # Return top RELEVANT product
    relevant_products = result.get('relevant', [])
    return relevant_products[0]  # Best match
```

Advantage: Uses same pipeline as regular search, maintains context
