BUDGET OPTIMIZER - GREEDY ALGORITHM
====================================

## Overview

Intelligent budget optimizer for task-based shopping that assembles product kits
within budget constraints using a greedy scoring algorithm with lookahead.

## Algorithm

### Phase 1: Product Scoring
Score all products using weighted formula:
  score = 0.5 × normalized_rating + 0.3 × semantic_score + 0.2 × price_efficiency

Where:
  - normalized_rating = rating / 5 (0 if missing)
  - semantic_score = hybrid search similarity (0.5 default)
  - price_efficiency = 1 - (price / max_price)

### Phase 2: Mandatory Item Selection (with Lookahead)
For each mandatory item:
  1. Calculate minimum cost for remaining mandatory items
  2. Reserve budget for remaining items
  3. Select highest scoring product that fits in available budget
  4. If no product fits, fail gracefully

Example:
  Budget: ₹4000
  Mandatory: [Shoes, Backpack]
  
  When selecting Shoes:
    - Need to reserve ₹1200 for cheapest backpack
    - Available for shoes: ₹4000 - ₹1200 = ₹2800
    - Select best shoe ≤ ₹2800
  
  This prevents selecting expensive shoe and running out of budget for backpack.

### Phase 3: Optional Item Selection (by Utility)
After mandatory items:
  1. Calculate utility for each optional product: utility = score / price
  2. Sort by utility (descending)
  3. Greedily add items with highest utility that fit budget
  4. Skip if budget insufficient

## Time Complexity

O(categories × top_k × log(top_k))
  - O(categories × top_k) for scoring
  - O(categories × top_k × log(top_k)) for sorting
  - O(categories × top_k) for selection

Much faster than brute force O(top_k^categories)

## Examples

### Example 1: High Budget (₹15,000)
Input:
  Activity: Hiking
  Budget: ₹15,000
  Mandatory: Shoes, Backpack
  Optional: Water Bottle, Trekking Poles

Output:
  ✓ Shoes: Mid-Range (₹3500, score=0.643)
  ✓ Backpack: Medium (₹2500, score=0.709)
  ✓ Water Bottle: Basic (₹150, utility=4.97)
  ✓ Poles: Basic (₹600, utility=1.23)
  Total: ₹6750, Remaining: ₹8250

### Example 2: Tight Budget (₹4,000)
Input:
  Activity: Hiking
  Budget: ₹4,000
  Mandatory: Shoes, Backpack
  Optional: Water Bottle, Trekking Poles

Output:
  ✓ Shoes: Budget (₹2000, score=0.627) [lookahead reserved ₹1200]
  ✓ Backpack: Basic (₹1200, score=0.686)
  ✓ Water Bottle: Basic (₹150, utility=4.97)
  ✓ Poles: Basic (₹600, utility=1.23)
  Total: ₹3950, Remaining: ₹50

### Example 3: Insufficient Budget (₹2,000)
Input:
  Activity: Hiking
  Budget: ₹2,000
  Mandatory: Shoes (min ₹2000), Backpack (min ₹1200)

Output:
  ❌ Failed: Budget too low for mandatory item: Hiking Shoes
  Message: "Budget too low for mandatory item: Hiking Shoes"

## Key Features

✅ **NO Brute Force** - Greedy selection, not exhaustive search
✅ **NO LLM** - Pure algorithmic approach
✅ **NO Recursion** - Simple iterative loops
✅ **Lookahead** - Reserves budget for remaining mandatory items
✅ **Deterministic** - Same input always produces same output
✅ **Fast** - O(n log n) complexity
✅ **Graceful Failure** - Clear error messages when budget insufficient

## Implementation Details

File: backend/tools/budget_optimizer.py

Key methods:
  - optimize(items, budget) - Main entry point
  - _score_all_products(items) - Scores all products
  - _score_product(product, max_price) - Individual product scoring
  - _select_mandatory_product(item, budget) - Greedy selection with lookahead
  - _calculate_utility(product) - Utility for optional items

Scores stored in: self._product_scores dictionary (indexed by product_id)

## Configuration

Scoring weights (adjustable):
  RATING_WEIGHT = 0.5      # Emphasizes customer satisfaction
  SEMANTIC_WEIGHT = 0.3    # Relevance to search query
  PRICE_WEIGHT = 0.2       # Value for money

Tight budget threshold:
  TIGHT_BUDGET_THRESHOLD = 0.7  # If min_cost > 70% of budget

## Edge Cases Handled

1. **No products for mandatory item** → Fail with clear message
2. **All products exceed budget** → Fail gracefully
3. **Zero or negative prices** → Handled (utility = 0)
4. **Missing ratings** → Use 0.0 as normalized rating
5. **Missing semantic scores** → Use 0.5 default
6. **Equal utility products** → Stable sort (order preserved)

## Comparison: Greedy vs Brute Force

Greedy (Current):
  - Time: O(n log n)
  - Memory: O(n)
  - Optimal: No (good approximation)
  - Speed: ~5ms for 100 products
  
Brute Force:
  - Time: O(k^n) where k=products, n=categories
  - Memory: O(k^n)
  - Optimal: Yes
  - Speed: ~5 seconds for 5^10 (9.7M combinations)

For hackathon MVP, greedy is the right choice.
