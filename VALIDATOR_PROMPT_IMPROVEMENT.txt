================================================================================
VALIDATOR PROMPT IMPROVEMENT - TIGHTER RELEVANT vs RELATED
================================================================================

LOCATION: training/local_model_server.py
ENDPOINT: POST /validate-products

================================================================================
THE PROBLEM
================================================================================

SCENARIO: User searches for "golf club"

HYBRID SEARCH RETRIEVES:
✅ Golf club sets (correct)
✅ Golf drivers (correct)
✅ Golf stand bags (contain "carry 14 golf clubs")
✅ Golf gloves (used with golf clubs)
✅ Golf umbrellas (used on golf course)

This is EXPECTED behavior from keyword + semantic search.

The validator is supposed to separate:
- RELEVANT: Actual golf clubs
- RELATED: Accessories (bags, gloves, covers)

BUT: The validator was being too permissive, marking bags as RELEVANT instead of
RELATED because they "relate to" golf clubs.

================================================================================
THE SOLUTION
================================================================================

TIGHTEN THE VALIDATOR'S DEFINITION

Don't change retrieval (it's working correctly).
Don't change ranking (it's finding the right candidates).
Only improve the validator's understanding of RELEVANT vs RELATED.

================================================================================
PROMPT CHANGES
================================================================================

BEFORE (too vague):
-----------------
Decisions:
RELEVANT = exact match or direct synonym
RELATED = accessory/attachment FOR this product type
NOT_RELEVANT = different product or unrelated

AFTER (clearer + examples):
---------------------------
Decision Rules:
RELEVANT: The candidate IS the requested product type.
RELATED: The candidate is an accessory, attachment, storage item, replacement 
         part, or complementary product used WITH the requested product.
NOT_RELEVANT: Different product type.

Example (Requested: golf club):
  Golf club set → RELEVANT
  Golf driver → RELEVANT
  Golf stand bag → RELATED (storage FOR clubs)
  Golf glove → RELATED (worn WHILE using clubs)
  Golf umbrella → RELATED (used ON golf course)
  Tennis racket → NOT_RELEVANT

Example (Requested: backpack):
  Hiking backpack 30L → RELEVANT
  Backpack rain cover → RELATED (protects backpack)
  Water bottle → RELATED (carried IN backpack)
  Tent → NOT_RELEVANT

================================================================================
WHY THIS WORKS
================================================================================

1. CLEARER DISTINCTION
   "IS the requested product" vs "used WITH the requested product"
   
   This makes it obvious that:
   - A bag is not a golf club (even if it carries clubs)
   - A glove is not a golf club (even if worn while playing)

2. CONCRETE EXAMPLES
   LLMs respond extremely well to examples.
   
   Showing "golf stand bag → RELATED" teaches the model that storage items
   are accessories, not the product itself.
   
   Two examples (golf club + backpack) generalize to other product types.

3. PARENTHETICAL EXPLANATIONS
   "(storage FOR clubs)" makes the reasoning explicit.
   
   This helps the model understand WHY something is RELATED vs RELEVANT.

================================================================================
IMPACT ON USER EXPERIENCE
================================================================================

Frontend naturally shows:
1. PRIMARY RESULTS (RELEVANT)
   - Actual golf clubs the user is searching for
   
2. RELATED PRODUCTS (RELATED)
   - Accessories and complementary items
   - Shown separately or lower in ranking

This matches user expectations:
- "I searched for golf club, show me golf clubs first"
- "Accessories are useful but secondary"

================================================================================
WHY NOT CHANGE RETRIEVAL?
================================================================================

Retrieval is working CORRECTLY.

When a user searches "golf club":
- They want to see golf clubs (primary)
- They might also want accessories (secondary)

Retrieval should find EVERYTHING related to golf clubs.
The validator then SEPARATES them into primary vs accessories.

If we filtered out bags/gloves at retrieval time:
❌ User searches "golf club" → can't discover useful accessories
❌ Loses valuable cross-sell opportunities
❌ Reduces product discovery

By keeping broad retrieval + smart validation:
✅ User sees golf clubs first (RELEVANT)
✅ User can discover accessories (RELATED)
✅ Frontend has flexibility to show/hide RELATED items
✅ Better user experience overall

================================================================================
VALIDATION RULES SUMMARY
================================================================================

RELEVANT:
- Product IS the requested type
- Example: "golf club" → golf club set, driver, putter

RELATED:
- Accessory: golf glove, golf hat
- Storage: golf stand bag, club cover
- Attachment: club grip, shaft
- Replacement part: club head
- Complementary: golf umbrella, rangefinder

NOT_RELEVANT:
- Different product type entirely
- Example: "golf club" → tennis racket, football

The key test: "IS this a [requested product]?" or "is it USED WITH [requested product]?"
