COMPARISON TOOL: DETERMINISTIC KEYWORD-BASED RESOLVER
======================================================

DATE: 2026-06-26
IMPLEMENTATION: Comparison-specific deterministic product resolver

DESIGN PRINCIPLE
----------------
The Compare Tool requires DETERMINISTIC ENTITY RESOLUTION, not semantic search.

Comparison must match EXACT products that exist in the knowledge base.
Users expect: "Compare A vs B" → Compare exactly A and exactly B.

Product Discovery is for semantic search and recommendations.
Compare Tool is for exact entity matching.

These are fundamentally different use cases and require different resolution strategies.

IMPLEMENTATION
--------------

Created a comparison-specific resolver isolated to Compare Tool only.
Does NOT use or modify Product Discovery pipeline.

Key Components:

1. _resolve_product_via_discovery()
   - Deterministic keyword-based matching
   - Operates only on products.name field
   - Prioritizes exact matches and model identifiers
   - No semantic search, no hybrid ranking

2. _normalize_product_name()
   - Case-insensitive normalization
   - Removes punctuation (preserves hyphens/plus signs in model numbers)
   - Normalizes whitespace
   - Creates consistent comparison format

3. _score_product_match()
   - Deterministic scoring based on keyword overlap
   - Prioritizes model identifiers (e.g., KD900X, KS900)
   - Brand + model matching
   - Returns score 0.0 - 1.0

RESOLUTION ALGORITHM
--------------------

For each product mention:

Step 1: Normalize mention
  - "Kiprun KD900X LD+" → "kiprun kd900x ld+"
  
Step 2: Extract model identifiers
  - Pattern: [A-Z]{2,}[0-9]+[A-Z]*
  - "Kiprun KD900X LD+" → ["KD900X", "LD"]
  
Step 3: Extract potential brand
  - First word: "Kiprun"
  
Step 4: Fetch candidates
  - Get products containing any keyword from mention
  - Limit to 100 candidates (performance optimization)
  
Step 5: Score each candidate
  Priority scoring:
  1. Exact normalized name match → 1.0
  2. High overlap (>70%) → 0.95
  3. All model identifiers match → 0.9
  4. Brand + keyword overlap >50% → 0.7-0.8
  5. Partial model match → 0.6-0.8
  6. Brand + weak overlap → 0.4-0.6
  
Step 6: Return highest scoring product

SCORING HIERARCHY
-----------------

Score  | Match Type                              | Example
-------|-----------------------------------------|------------------
1.00   | Exact normalized name match             | "kiprun kd900x" == "kiprun kd900x"
0.95   | High overlap (>70%)                     | "kiprun kd900x ld+" matches "kiprun kd900x ld+ running shoes"
0.90   | All model identifiers match             | KD900X found in product name
0.70   | Brand + strong keyword overlap (>50%)   | "kiprun" + 60% keywords match
0.60   | Partial model identifier match          | Some model IDs match
0.40   | Brand + weak keyword overlap (30-50%)   | "kiprun" + 40% keywords match
0.00   | No match                                | No overlap

DIFFERENCES FROM PRODUCT DISCOVERY
-----------------------------------

Product Discovery (HybridSearchService):
- Purpose: Semantic product search and recommendations
- Method: Hybrid ranking (semantic + keyword + validation)
- Scoring: Combines similarity, validation confidence, profile
- Use case: "Find waterproof backpacks" → broad search
- Pipeline: SQL → Semantic search → Ranking → Validation

Compare Tool Resolver:
- Purpose: Deterministic entity resolution
- Method: Keyword-based exact matching
- Scoring: Keyword overlap + model identifiers
- Use case: "Compare Kiprun KD900X vs KD500" → exact match
- Pipeline: Normalize → Fetch candidates → Score → Select best

KEY DIFFERENCES:
1. No semantic embeddings
2. No LLM validation
3. No hybrid ranking
4. Deterministic scoring only
5. Exact match prioritization

PERFORMANCE
-----------

Optimizations:
- Fetches only candidates containing at least one keyword (not all products)
- Limits to 100 candidates max
- No semantic embedding computation
- No LLM validation calls
- Simple keyword matching (fast)

Typical performance:
- 2 products to resolve: 10-50ms total
- No external API calls
- Database-only operation

EDGE CASES
----------

1. Direct Product ID
   - If mention is a product ID → return immediately
   - No scoring needed

2. No candidates found
   - Returns "Product not found in the knowledge base"
   - Clear error message

3. No matches above threshold
   - Returns error with product mention
   - Suggests checking product name

4. Ambiguous matches
   - Returns highest scoring match
   - Logs top 5 for debugging

5. Model identifier variations
   - Handles: KD900X, KD-900X, KD 900X
   - Normalization preserves hyphens and plus signs

ADVANTAGES
----------

1. Deterministic
   - Same input always gives same output
   - No semantic ambiguity
   - Predictable behavior

2. Fast
   - No embeddings computation
   - No LLM calls
   - Simple keyword matching

3. Accurate for entity resolution
   - Prioritizes exact matches
   - Respects model identifiers
   - Brand-aware

4. Isolated
   - Compare Tool only
   - Doesn't affect Product Discovery
   - No side effects

5. Maintainable
   - Simple logic
   - Easy to debug
   - Clear scoring rules

LIMITATIONS
-----------

1. Requires products exist in database
   - Cannot compare products not in knowledge base
   - This is by design (entity resolution, not semantic search)

2. Keyword-based only
   - No understanding of synonyms
   - "running shoes" != "trainers"
   - This is acceptable for comparison use case

3. Not suitable for discovery
   - Would not work for "find waterproof backpacks"
   - Only for exact entity matching
   - Product Discovery is correct tool for discovery
