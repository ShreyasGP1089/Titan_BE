COMPARISON TOOL: TOLERANT TOKEN-BASED MATCHING
===============================================

DATE: 2026-06-26
IMPLEMENTATION: Deterministic but tolerant matching with partial results

DESIGN PHILOSOPHY
-----------------
✅ Deterministic (no semantic embeddings, no LLM, no hallucinations)
✅ Tolerant (normalized tokens, not exact string equality)
✅ Graceful degradation (partial results instead of complete failure)

SCORING SYSTEM
--------------
Score out of 100:

100: Exact normalized name match
     "size 5 machine stitched football training ball white"
     ==
     "size 5 machine stitched football training ball white"

80:  All significant tokens match (≥90% overlap)
     Query tokens: {size, 5, machine, stitched, football, training, ball, white}
     Product tokens: {size, 5, machine, stitched, football, training, ball, white, official}
     Overlap: 8/8 = 100% → Score 80

70:  High token overlap (≥80%) with model IDs
     Query tokens: {kiprun, kd900x, ld, running, shoes}
     Product tokens: {kiprun, kd900x, ld, trail, running, shoes}
     Overlap: 5/5 = 100% → Score 70

60:  Good token overlap (≥70%) with model IDs
     Overlap: 4/5 = 80% → Score 60

50:  Moderate overlap (≥60%) with model IDs
     Overlap: 3/5 = 60% → Score 50

40:  Some overlap (≥50%) with model IDs
     Overlap: 3/6 = 50% → Score 40 (MINIMUM THRESHOLD)

<40: Below threshold → Not a match

MANDATORY RULES
---------------

1. Model Identifiers are MANDATORY
   If query contains model IDs (KD900X, MT500, etc):
   → ALL must be present in product name
   → Missing ANY model ID → Score 0

2. Threshold
   Score must be ≥40 to be considered a match
   Below 40 → Product not found

3. Stop Words Removed
   Ignored: the, a, an, and, or, in, on, at, to, for, of, with, etc.
   These don't count toward token overlap

PARTIAL RESULTS
---------------

Instead of throwing exceptions, return partial results:

CompareResponse {
  products: [Product1, Product2],      // Successfully resolved
  missing_products: ["Product3"],      // Could not resolve
  message: "1 of 3 products could not be resolved from the knowledge base."
}

Benefits:
- Frontend can show nicer error messages
- If only one product failed, user still sees what was found
- Better UX than complete failure

NORMALIZATION
-------------

Process:
1. Convert to lowercase
2. Remove punctuation (except hyphens/plus in model codes)
3. Normalize whitespace
4. Extract tokens
5. Remove stop words

Examples:
"Size 5 Machine-Stitched Football Training Ball - White"
→ "size 5 machine stitched football training ball white"

"Kiprun KD900X LD+ Running Shoes"
→ "kiprun kd900x ld+ running shoes"

TOKEN EXTRACTION
----------------

Significant tokens:
- Length > 1 character
- Not in stop words list
- Alphanumeric

Stop words removed:
the, a, an, and, or, but, in, on, at, to, for, of, with, by, from, etc.

Example:
"The Size 5 Ball for Training"
→ Tokens: {size, 5, ball, training}  (removed: the, for)

COMPARISON WITH PREVIOUS VERSIONS
----------------------------------

V1 (Semantic Search):
- Used Product Discovery pipeline
- Semantic embeddings + LLM validation
- Could match unrelated products (hallucinations)
- ❌ Too flexible, not deterministic

V2 (Exact String Matching):
- Required exact normalized string equality
- Threw exceptions immediately
- ❌ Too strict, brittle

V3 (Token-Based Matching) ← CURRENT:
- Deterministic token overlap scoring
- Tolerant to formatting variations
- Partial results for graceful degradation
- ✓ Balanced: deterministic but tolerant

EDGE CASES
----------

1. Query: "Size 5 Ball" (simple, no model ID)
   - Uses token overlap
   - "Size 5 Football Ball" → 100% overlap → Score 80 ✓

2. Query: "KD900X" (model ID only)
   - Model ID mandatory
   - "Kiprun KD900X Shoes" → Has KD900X → Score 60 ✓
   - "Kiprun Shoes" → Missing KD900X → Score 0 ✗

3. Query: "XYZ999 Ball" (non-existent model)
   - Model ID mandatory
   - No products have XYZ999
   - All score 0
   - Returns: "Product not found" ✓

4. Query: "Football Ball" (generic)
   - No model IDs
   - Many products match
   - Returns: Highest scoring product (token overlap) ✓

5. Two products, one not found
   - Product 1: Found ✓
   - Product 2: Not found ✗
   - Returns: Partial results with message ✓

THRESHOLD RATIONALE
-------------------

Why 40/100?
- Below 50% token overlap is too weak for comparison
- We want deterministic matching, not guessing
- Better to say "not found" than return wrong product

Could be adjusted based on use case:
- Stricter: Threshold 60 (only strong matches)
- More lenient: Threshold 30 (allow weaker matches)

Current (40) is balanced: allows formatting variations but rejects weak matches.
