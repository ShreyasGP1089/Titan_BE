COMPARISON TOOL: QUERY-RELATIVE TOKEN SCORING
==============================================

DATE: 2026-06-26
UPDATE: Query-relative overlap with graceful numeric mismatches

PROBLEM WITH PREVIOUS APPROACH
-------------------------------
Product names contain many descriptive tokens:
- "Football Ball Training Size 4 Age 8 to 12 years First Kick Red"

Query is simpler:
- "Football Ball Size 5 First Kick"

OLD calculation:
  matched_tokens = {football, ball, first, kick}  // 4 tokens
  query_tokens = {football, ball, size, 5, first, kick}  // 6 tokens
  product_tokens = {football, ball, training, size, 4, age, 8, to, 12, years, first, kick, red}  // 13 tokens
  
  overlap_ratio = 4 / 6 = 67%  // Seemed OK
  
But "size" was in product, so actually matched 5/6 = 83%
However, the SIZE VALUE differed (5 vs 4), causing confusion.

NEW APPROACH
------------

1. Query-Relative Overlap
   - matched = query_tokens ∩ product_tokens
   - coverage = matched / query_tokens
   - Extra product tokens DON'T reduce score

2. Separate Numeric Handling
   - Model ID numbers: Mandatory (part of model identifier)
   - Descriptive numbers: Soft match (size, age, etc.)
   - Numeric mismatch: -10 penalty, not discard

3. Token Categories
   - Mandatory: Model IDs (must match exactly)
   - Important: Head nouns, key descriptors
   - Optional: Colors, ages, marketing words

SCORING FORMULA
---------------

Step 1: Check mandatory model identifiers
  if model_identifiers:
    if ANY missing → return 0

Step 2: Calculate query coverage
  matched = query_tokens ∩ product_tokens
  coverage = len(matched) / len(query_tokens)

Step 3: Base score from coverage
  100% coverage → 90 points
  90%+ coverage → 80 points
  80%+ coverage → 70 points
  70%+ coverage → 60 points
  60%+ coverage → 50 points
  50%+ coverage → 40 points (minimum)
  <50% coverage → 0 points

Step 4: Apply numeric mismatch penalty
  if descriptive_numbers_differ:
    score -= 10

Step 5: Model ID bonus
  if model_ids_present_and_matched:
    score += 5

NUMERIC HANDLING
----------------

Category 1: Model ID Numbers (MANDATORY)
  Examples: "KD900X", "MT500", "NH100"
  Rule: Must match exactly
  Missing → Score 0

Category 2: Descriptive Numbers (SOFT)
  Examples: Size, Age, Quantity
  Rule: Mismatch → -10 penalty
  Missing → No penalty

Detection:
  Model ID numbers = numbers within model identifiers
  Descriptive numbers = other numbers in query/product

Example:
  Query: "Kiprun KD900X Size 5"
  - "900" is part of KD900X → Model ID number (mandatory)
  - "5" is a size → Descriptive number (soft match)

BENEFITS
--------

1. Tolerant to Product Variations
   - "Size 5 Ball" matches "Size 4 Ball Training Red"
   - Extra descriptive words don't hurt score
   - Realistic matching for catalog data

2. Numeric Flexibility
   - Size 5 query can match Size 4 product (with penalty)
   - Better than "not found" if no exact size exists
   - Model IDs still enforced strictly

3. Query-Centric Scoring
   - User's query defines what's important
   - Product can have extra details
   - Matches user mental model

4. Predictable Degradation
   - 100% coverage → 90 points
   - Each missing query token → lower score
   - Numeric mismatch → clear -10 penalty

   Numeric mismatch: 3 vs 5 (quantity) → -10

3. Color mismatch (not numeric)
   Query: "Football Ball Red"
   Product: "Football Ball Blue"
   
   Matched: {football, ball} = 2/3 = 67%
   Score: 50
   Result: Moderate match (color differs but not penalized as numeric)

THRESHOLD RATIONALE
-------------------

Minimum: 40 points (50% query coverage)
- Below 50% means too many query tokens missing
- Better to return "not found" than weak match
- Maintains deterministic quality

With penalties:
- 50% coverage - 10 (numeric) = 40 (at threshold)
- 60% coverage - 10 (numeric) = 50 (acceptable)
- 70% coverage - 10 (numeric) = 60 (good)
